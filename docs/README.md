# OctoAcme Project Management Docs

This README provides a concise summary of OctoAcme's project management processes and direct links to the detailed process documents stored in the docs/ folder. It is intended as a landing page to improve discoverability and reduce onboarding friction for contributors and stakeholders.

## Brief summary

OctoAcme follows a lightweight, iterative project management approach that emphasizes customer value, clear ownership, and measurable outcomes. Projects begin with a focused initiation (one‑pager, stakeholders, success metrics, and a go/no‑go decision gate). Approved initiatives move into planning where work is broken into shippable increments, acceptance criteria and a Definition of Done are set, dependencies and risks are captured, and a release plan with milestones is created.

During execution teams track progress using prioritized backlogs and regular cadenced communication: PM + PdM weekly syncs, delivery team standups, and monthly stakeholder updates. Backlog items include acceptance criteria and an initial QA approach; developers write and maintain tests and documentation, and QA validates acceptance criteria prior to release. Releases follow a deploy/verify/announce pattern and are followed by retrospectives to capture learnings and action items.

Risk management and communication are explicit: risks are recorded in a Risk Register with owners and mitigations, cross‑team dependencies are surfaced on project boards, and escalation paths are defined. Roles and personas (Product Managers, Project Managers, Developers, QA, and Stakeholders) are defined with clear responsibilities to ensure alignment and accountability.

## How to use this folder
- Keep the one‑pager, charter, and process artifacts up to date in this repo under docs/.
- Add project‑specific process docs to `.copilot/` if you want Copilot Spaces to use them as context for exercises.
- Use the checklists in each doc during initiation, planning, execution, and retrospectives to ensure consistent delivery.
